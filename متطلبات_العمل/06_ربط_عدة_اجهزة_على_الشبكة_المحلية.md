# 06) ربط 4 أجهزة على شبكة محلية (LAN) — مهم جدًا

## 1) الوضع الحالي للمشروع (اليوم)

التطبيق حاليًا:

- يعمل **محليًا على كل جهاز**
- قاعدة البيانات: SQLite محلية  
  `Documents\Rose_School_edu\data\rose_school_2026.db`
- الملفات (صور/مرفقات) محلية على نفس الجهاز
- تسجيل الدخول موجود (مستخدم/كلمة مرور) لكنه **محلي لنسخة الجهاز**
- **لا يوجد سيرفر مركزي**
- **لا يوجد تزامن حي** بين الأجهزة

### ماذا يعني عمليًا؟
إذا ثبّتت البرنامج على 4 أجهزة:
- كل جهاز له بيانات مستقلة
- إضافة طالب على جهاز A **لا تظهر** فورًا على B/C/D
- خطر التعارض إذا حاولتم مشاركة ملف SQLite يدويًا عبر الشبكة

> لذلك: المطلوب الذي وصفته (4 أجهزة + حفظ متزامن + مستخدم مستقل) **غير مدعوم بالكامل في النسخة الحالية**، ويحتاج تطوير طبقة شبكة/سيرفر.

---

## 2) هدفك (المطلوب النهائي)

- 4 أجهزة Desktop/Laptop على **شبكة محلية واحدة**
- كل موظف يدخل بـ:
  - **اسم مستخدم مستقل**
  - **كلمة مرور مستقلة**
- الجميع يشتغل على **نفس البيانات**
- إضافة/تعديل/حفظ **بدون تضارب**
- عند انقطاع جهاز، لا تُفسد قاعدة البيانات

---

## 3) الحل الصحيح (موصى به للمدارس)

### المعمارية: جهاز رئيسي (Server) + 3/4 أجهزة عملاء (Clients)

```text
[جهاز السيرفر]  <--- LAN --->  [جهاز 1]
       ^                         [جهاز 2]
       |                         [جهاز 3]
       +-----------------------> [جهاز 4]
```

### جهاز السيرفر (واحد فقط)
- يستضيف قاعدة البيانات المركزية
- يستقبل طلبات الإضافة/التعديل/القراءة
- يدير الصلاحيات والمستخدمين
- يحفظ الملفات/المرفقات مركزيًا (أو مسار شبكي منظم)

### أجهزة العملاء
- تطبيق Rose School
- تسجيل دخول مستخدم/كلمة مرور
- لا تحفظ قاعدة مستقلة كمصدر حقيقة
- ترسل/تستقبل عبر الشبكة المحلية

---

## 4) لماذا لا نشارك ملف SQLite مباشرة على الشبكة؟

بعض الناس يفكرون:
> نضع ملف `.db` على مجلد مشترك ويعمل الجميع عليه

**هذا خطر وغير مناسب** لأن:
- SQLite ليس مصممًا لتعدد مستخدمين كتابيين عبر SMB/Network Share
- احتمالية تلف القاعدة عالية
- أخطاء قفل/تعارض حفظ شائعة
- صعوبة الاستعادة عند الانقطاع

### الحكم
- ❌ لا لمشاركة ملف SQLite الخام بين 4 أجهزة كحل إنتاج
- ✅ نعم لسيرفر مركزي (API/DB server)

---

## 5) الخيارات التقنية الممكنة

### الخيار A (الأفضل تدريجيًا لمشروعك الحالي):  
**Local Server داخل التطبيق / بجانبه**

- جهاز السيرفر يشغّل:
  - تطبيق Rose + خدمة محلية (API)
  - أو خدمة مستقلة صغيرة
- باقي الأجهزة تتصل بـ IP السيرفر مثل:
  - `http://192.168.1.10:8080`

**إيجابيات**
- يبقى كل شيء داخل المدرسة (بدون إنترنت)
- تحكم كامل بالصلاحيات
- حفظ منظم وتعارض أقل

**سلبيات**
- يحتاج تطوير (متوسط/كبير)
- السيرفر يجب أن يكون شغالًا أثناء العمل

---

### الخيار B (أقوى طويل الأمد):  
**PostgreSQL / MySQL مركزي + API**

- قاعدة بيانات حقيقية متعددة المستخدمين
- تطبيق Flutter يتصل عبر API

**إيجابيات**
- أفضل استقرار مع الكتابة المتزامنة
- مناسب للتوسع

**سلبيات**
- إعداد أكثر
- يحتاج صيانة

---

### الخيار C (مؤقت فقط — غير مفضل):  
**جهاز واحد رئيسي + مزامنة نسخ احتياطي يدوية/مجدولة**

- العمل الأساسي على جهاز واحد
- أو كل جهاز يعمل فترة ثم دمج يدوي

**لا يحقق** “4 أجهزة معًا بدون مشاكل” فعليًا.

---

## 6) ما المطلوب تقنيًا لتحقيق هدفك “بدون مشاكل”

### أ) شبكة
- راوتر/سويتش محلي ثابت
- IP ثابت لجهاز السيرفر (مهم)
- كل الأجهزة على نفس الشبكة (نفس الـ subnet)
- جدار حماية ويندوز يسمح بمنفذ الخدمة (مثل 8080)

### ب) حسابات وصلاحيات
لكل مستخدم:
- username
- password (مشفّر/hashed)
- الدور: إدارة / أمانة سر / محاسبة / امتحانات...
- صلاحيات أبواب محددة

### ج) قاعدة بيانات مركزية
جداول/سجلات موحدة:
- students
- employees
- invoices/receipts
- dues/notifications
- users/roles
- audit log (من عدّل ماذا ومتى)

### د) منع التعارض (Concurrency)
- كل تعديل يمر عبر السيرفر
- قفل منطقي للسجلات الحساسة (اختياري)
- أو “آخر كتابة رابحة” مع سجل تدقيق
- معاملات DB (transactions) للعمليات المالية

### هـ) الملفات (صور/مرفقات)
- تخزين مركزي على السيرفر
- أو مجلد شبكي `\\SERVER\RoseFiles` مع مسارات محفوظة في DB
- ممنوع مسارات محلية مطلقة لكل جهاز بشكل مختلف دون توحيد

### و) النسخ الاحتياطي
- Backup يومي تلقائي على السيرفر
- نسخة ثانية على قرص خارجي/جهاز آخر

---

## 7) كيف يدخل كل جهاز بمستخدم مستقل؟

نعم هذا مطلوب وسهل ضمن السيرفر:

1. جدول Users مركزي
2. شاشة Login على كل جهاز
3. بعد النجاح:
   - تحميل صلاحيات المستخدم
   - إخفاء/إظهار الأبواب حسب الدور
4. كل عملية حفظ تُسجَّل باسم المستخدم

> عندكم أساس محلي للمستخدمين/الصلاحيات الآن، لكن يجب نقله ليصبح مركزيًا.

---

## 8) سيناريو تشغيل يومي مقترح (4 أجهزة)

1. تشغيل **جهاز السيرفر** أولًا
2. التأكد أن خدمة البيانات تعمل
3. تشغيل الأجهزة الثلاثة/الأربعة
4. كل موظف يدخل بحسابه
5. العمل على نفس البيانات
6. نهاية الدوام: Backup تلقائي/يدوي من السيرفر

---

## 9) خطة تنفيذ مقترحة (مراحل)

### المرحلة 0 — الآن (بدون تطوير كبير)
- جهاز رئيسي واحد للعمل الرسمي
- باقي الأجهزة للعرض/الطباعة فقط
- أو نسخ احتياطي مجدول (حل مؤقت)

### المرحلة 1 — شبكة جاهزة
- IP ثابت للسيرفر
- اختبار ping بين الأجهزة
- تحديد من السيرفر

### المرحلة 2 — سيرفر بيانات محلي
- API محلي للقراءة/الكتابة
- نقل المستخدمين والصلاحيات مركزيًا
- تسجيل دخول مركزي

### المرحلة 3 — العمليات الحرجة أولًا
نبدأ بمزامنة:
1. الطلاب
2. الأقساط/الدفعات/المستحقين
3. الإيرادات/الصرفيات
ثم باقي الأبواب

### المرحلة 4 — تعارضات وموثوقية
- قفل/تدقيق
- إعادة اتصال تلقائي
- رسائل “السيرفر غير متصل”

### المرحلة 5 — تثبيت وإنتاج
- Setup للعملاء
- Setup/خدمة للسيرفر
- تدريب المستخدمين

## 10) تقدير الجهد (تقريبي)

| البند | الجهد |
|------|------|
| تجهيز الشبكة + IP ثابت | صغير |
| Login مركزي + صلاحيات | متوسط |
| API + قاعدة مركزية للبيانات الأساسية | متوسط إلى كبير |
| كل الأبواب + الملفات + التعارضات | كبير |
| اختبار 4 أجهزة حقيقي | ضروري |

## 11) قرارات يجب أن تؤخذ قبل البرمجة

أجب عليها في الجلسة القادمة:

1. **من جهاز السيرفر؟**
   - جهاز إدارة ثابت طوال الدوام؟
2. **هل السيرفر يُستخدم أيضًا كجهاز عمل عادي؟**
3. **هل العمل يجب أن يستمر إذا انقطع السيرفر؟**
   - (Offline mode أعقد بكثير)
4. **هل تريد صلاحيات دقيقة لكل باب؟**
5. **هل المرفقات/الصور ضرورية على كل الأجهزة فورًا؟**
6. **عدد المستخدمين المتوقع** (4 أجهزة ≠ 4 مستخدمين دائمًا)

---

## 12) توصيتي المباشرة لك

### للتثبيت الآن بدون مخاطرة
- اجعل **جهازًا واحدًا** هو مصدر الحقيقة
- لا تشارك ملف SQLite بين الأجهزة

### للهدف الحقيقي (4 أجهزة معًا)
- اعتمد **Server + Clients على LAN**
- مستخدم/كلمة مرور مركزية
- حفظ عبر السيرفر فقط

### لا أنصح
- مشاركة `rose_school_2026.db` من مجلد شبكي كحل دائم

---

## 13) ماذا أحتاج منك لأبدأ التنفيذ البرمجي؟

أرسل:
1. هل يوجد جهاز يمكن تركه شغالًا كسيرفر طوال الدوام؟
2. أسماء/أدوار المستخدمين الأربعة (إدارة، أمانة، محاسبة...)
3. هل تقبل أن يتوقف العمل عند إطفاء السيرفر؟
4. هل نبدأ أولًا بـ:
   - الطلاب + المحاسبة فقط  
   أم كل النظام دفعة واحدة؟

---

## 14) خلاصة جملة واحدة

> لربط 4 أجهزة مع حفظ آمن ومستخدمين مستقلين: نحتاج **قاعدة/خدمة مركزية على جهاز سيرفر**، وليس مجرد نسخ البرنامج على كل جهاز.
